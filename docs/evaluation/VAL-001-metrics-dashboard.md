## 1. 指标框架分层 (Metrics Framework Hierarchy)

根据具身智能机器人在真实物理环境下的作业特性，我们将监控指标划分为北极星指标、输入指标、健康护栏指标和业务指标四个层级。

### 北极星指标 (North Star Metric)
- **平均人工干预间隔时间 (Mean Time Between Overrides, MTBO)**
  - *定义理由*：直接反映机器人任务执行的自主度（Autonomy）。MTBO 越长，人机配比率越低，说明商业化落地的性价比与可行性越高。

### 输入指标 (Input Metrics)
- **任务成功率 (Task Success Rate, TSR)**：高层逻辑拆解与技能调度成功的占比。
- **路径执行效率比 (Path Efficiency Ratio, PER)**：实际轨迹对比理论最优规划的绕路冗余度。
- **接管转换时延 (Teleop Transition Latency)**：从端侧安全拦截上报，到云端操作员接管通路建立的时延。
- **视觉分割置信度 (Perception Confidence)**：VLM 模型对环境物体及物理边界识别的置信区间。

### 健康指标 (Health Metrics)
- **安全反射误拦截率 (Safety False Positive Rate)**：安全引擎因算法误判（如将阴影判定为障碍物）触发 ESTOP 的概率。
- **端云通信丢包率 (Network Packet Loss Rate)**：WebSocket 遥测时序流在端云间的网络丢包率。
- **关节电机过温/过流发生率 (Motor Thermal/Current Fault Rate)**：由于物理负荷过大或碰撞摩擦导致电机超出额定限制的频次。

### 业务指标 (Business Metrics)
- **单次任务能耗成本 (Task Energy Cost)**：机器人执行单次典型任务所消耗的电池容量电费与云端 API 调用费用。
- **操作员利用率 (Teleoperator Utilization Rate)**：云端操作员在班期间，实际执行远程控制的时间占比。
- **硬件平均故障间隔时间 (Mean Time Between Failures, MTBF)**：机器人关节减速箱、伺服电机等物理硬件损坏的物理寿命指标。

## 2. 指标定义矩阵 (Metric Definition Matrix)

| 指标 | 定义 | 数据来源 | 可视化类型 | 目标值 | 告警阈值 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **MTBO** | 自主运行总时长 / 人工接管次数（30天移动窗口） | 遥测服务日志 | 折线图 (天级) | >= 12.0 小时 | < 4.0 小时 |
| **TSR** | (成功完成任务指令数 / 总下发指令数) * 100% | 任务执行数据库 | 漏斗图 / 占比图 | >= 95.0% | < 85.0% |
| **PER** | 理论最短路径长度(m) / 实际移动轨迹长度(m) | 端侧里程计与规划器 | 分布散点图 | >= 0.85 | < 0.65 |
| **接管时延** | 接管指令生效时间戳 - 安全事件触发时间戳 | 控制台接入日志 | 柱状图 | <= 1.5 秒 | > 5.0 秒 |
| **安全误拦率** | 人工标定为误判的ESTOP数 / 总安全拦截数 | 坏案标注数据库 | 折线图 | <= 2.0% | > 8.0% |
| **通信丢包率** | (丢失的遥测帧数 / 应收的遥测帧总数) * 100% | WebSocket 监控 | 折线图 (分钟级)| <= 0.1% | > 1.0% |
| **单次任务能耗**| 单次 Session 消耗电量(Wh) + 云端算力费 (RMB) | 电池BOM与算力账单| 柱状图 | <= 1.2 元/次 | > 3.0 元/次 |

## 4. 运营评审节奏 (Review Cadence)

- **每日评审 (Daily Standup)**
  - *关注对象*：系统运维团队、值班操作员。
  - *关注指标*：健康指标（通信丢包率、关节温升异常、实时 ESTOP 警报日志）。
  - *行动目标*：快速排除单机硬件故障，解决当前网络接入异常。
- **每周评审 (Weekly Operational Review)**
  - *关注对象*：产品经理、算法工程主管。
  - *关注指标*：输入指标趋势（TSR、PER、接管时延、感知分割置信度分布）。
  - *行动目标*：拉齐上周坏案（Badcase）清单，跟进 Prompt 优化与安全拦截器阈值微调。
- **每月评审 (Monthly Business Review)**
  - *关注对象*：产品总监、商业化负责人。
  - *关注指标*：北极星指标（MTBO 长期走势）、业务指标（单次能耗成本、操作员利用率）。
  - *行动目标*：评估人机比是否达到盈亏平衡点，校准产品版本发布路线图与 OKR 达成进展。
- **每季度战略会 (Quarterly Strategic Review)**
  - *关注对象*：公司管理层、核心科学家。
  - *关注指标*：硬件 MTBF 趋势、行业安全性合规标准进展。
  - *行动目标*：评估技术路线调整（如是否从端云协同转向完全端侧边缘计算），重构指标体系。

## 5. 告警规则与分级响应策略 (Alert & Alarm Policies)

告警规则由端侧与云端监控服务自动运行，分级通知对应的负责人。

### 5.1 告警分级矩阵

| 告警级别 | 触发条件 | 接收渠道 | 期望响应时间 | 预警行动与负责人 |
| :--- | :--- | :--- | :--- | :--- |
| **Level 1 (P0-Emergency)** | 任意一台机器人的 `safety_event` 字段上报 `SEC-003` (电机过流过温) 或连续触发 `Critical` 拦截且自动恢复失败。 | 即时短信 + 钉钉/Slack 机器人高频 @ 运维主管与端侧控制组。 | < 5 分钟 | 底层控制开发人员介入，机器人自动断电，进入人工在环物理顺从状态。 |
| **Level 2 (P1-Critical)** | 单个工作日（24小时窗口内）全网机器人平均 `TSR` 跌破 80% 或 `MTBO` 跌破 3 小时。 | 邮件 + 飞书/钉钉群组预警，抄送算法负责人及产品经理。 | < 2 小时 | 大模型规划组与 PM 召开紧急坏案排查会议，拉取当前 Session 日志诊断是否由于大模型升级或 Prompt 变更导致规划故障。 |
| **Level 3 (P2-Warning)** | 过去 7 天累计通信丢包率超过 1.0% 或单任务能耗成本连续三天上涨 > 20%。 | 每日邮件日报汇总标注异常，通知基础网络开发组与商务运营组。 | < 24 小时 | 基础网络组排查特定小区的基站 5G 通信时延，商务组核算云端算力消耗异常。 |

## 6. 分析与监控工具链推荐 (Tooling Recommendations)

基于本系统的端云混合特性，我们推荐使用以下工具构建这一指标评测看板：

1. **端侧物理遥测 (Operational Telemetry)**：**Prometheus + Grafana**
   - *选用原因*：端侧高频的关节扭矩、温度、通信丢包属于时序数据（Time-series）。端侧安装 Node Exporter，将时序上报至 Prometheus，通过 Grafana 绘制机器人的秒级监控看板，非常适合“每日评审”。
2. **高层任务流分析 (Product & Task Flow Analytics)**：**PostHog**
   - *选用原因*：用户指令解析成行为树、发生接管、最终成功或失败，本质上是行为事件链。PostHog 支持漏斗分析、路径分析与坏案 Session Replay，非常适合“每周评审”的坏案分析与标注。
3. **商业数据看板 (Business BI)**：**Metabase**
   - *选用原因*：支持编写 SQL 灵活查询 PostgreSQL/BigQuery 中存储的历史会话元数据，能够直观统计月度/季度维度的 MTBO、单任务能耗以及操作员利用率趋势，易于产品经理（PM）进行指标汇报。
